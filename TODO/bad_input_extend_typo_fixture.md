# `bad_input` fixture `extend_typo` can never pass (tabs vs. spaces)

## Summary

`bad_input/type_errors/extend_typo/expected_output.txt` stores its field separators
as **spaces** where the reformatter emits **tabs**, so the fixture's `diff` can never
match. It is the **only** failing bad-input test, and it fails on a clean tree — this
is pre-existing, not caused by any in-flight work.

The practical cost: **`pre_commit_check.sh` always fails at step 6** ("Bad input tests
have failures"), so the one gate that is supposed to say "safe to commit" is
permanently red, and a *real* regression in any of the other 36 fixtures would be easy
to wave through as "oh, that's just the known one".

## Evidence

The runner compares byte-for-byte (`diff -q "$actual_file" "$expected_file"` —
[run_bad_input_tests.sh:88](../bad_input/run_bad_input_tests.sh#L88)), and
[normalize_output.lua](../bad_input/normalize_output.lua) does not touch interior
whitespace (it only strips timestamps, rewrites the temp dir to `{DIR}`, normalises
path separators, and trims *trailing* whitespace). So a tab/space mismatch is fatal.

Expand the tabs on both sides and they are identical:

```bash
diff <(expand -t 8 actual_normalized.txt) \
     <(expand -t 8 bad_input/type_errors/extend_typo/expected_output.txt)
# no output — whitespace is the ONLY difference
```

And this fixture is the odd one out — every sibling in the same category has tabs:

```text
bad_input/type_errors/bad_boolean/expected_output.txt             tabs=4
bad_input/type_errors/bad_enum_value/expected_output.txt          tabs=4
bad_input/type_errors/extend_typo/expected_output.txt             tabs=0   <-- this one
bad_input/type_errors/out_of_range_ubyte/expected_output.txt      tabs=5
bad_input/type_errors/string_for_integer/expected_output.txt      tabs=6
bad_input/type_errors/unknown_type_in_union/expected_output.txt   tabs=7
bad_input/type_errors/unknown_type_suggestion/expected_output.txt tabs=5
```

Almost certainly the file was hand-written or pasted through something that expanded
tabs (an editor with "insert spaces", or a copy out of a terminal), rather than being
generated by the runner's own `--update` mode.

## Fix

The expected *content* is correct — the five diagnostics are the right ones for this
input. Only the separators are wrong. So:

```bash
./bad_input/run_bad_input_tests.sh --update type_errors extend_typo
```

`--update` copies the freshly normalised actual output over `expected_output.txt`
([run_bad_input_tests.sh:75-80](../bad_input/run_bad_input_tests.sh#L75-L80)), which
writes the separators as real tabs.

**Review the resulting diff before committing it** — `--update` is "bless whatever the
code prints now", so it would silently bake in a genuine regression too. What you
should see is *only* whitespace changing on those five lines. (Do **not** blind-replace
spaces with tabs by hand: the messages contain legitimate interior spaces, e.g. `Bad
type description, col 3 in ...`.)

## Worth considering (optional)

Two ways to stop this recurring, if it seems worth the churn:

- Have `normalize_output.lua` **canonicalise the separator** (collapse runs of
  whitespace between the four log fields to a single tab) on both the actual and the
  expected side, so a hand-edited expected file with spaces still compares equal.
- Or add a **guard** to the runner: if an `expected_output.txt` contains no tab at all
  while the actual output does, report it as a *malformed fixture* rather than a
  content mismatch, so the next person is not left diffing two blocks of text that
  look character-for-character identical on screen. That "looks identical but FAILs"
  presentation is the real trap here.
