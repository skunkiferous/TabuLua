Note (added while implementing boxed_int64.md Phase 5e): an int64 column now exports as
BIGINT with a bare literal, e.g. ('ironSword',9007199254740993,...). That is exact in the
file and exact once stored in SQLite. The hazard is on the way OUT: on LuaJIT every Lua
number is a double, so reading such a column through tonumber -- or through any SQL
binding that returns a Lua number, such as lsqlite3 -- silently rounds anything past 2^53
(9007199254740993 comes back as ...992, with no error). Read the digits as TEXT instead:
CAST(col AS TEXT), or parse the literal's characters. This matters for the "load the
exported data in sqlite" idea above, and for any external tool doing the same.
See TODO/export_format_reimport.md (Scope) for the longer version.
